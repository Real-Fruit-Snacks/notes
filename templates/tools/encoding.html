Understanding the representations

It's all bytes

Every row on this page shows the same bytes through a different lens. Text is just one of the lenses — your input is decoded to bytes first (UTF-8 when you choose plain text), and Base64, hex, binary, percent-encoding and the digests are all computed from that one byte array. Change the mode, not the data.

Base64

Base64 repacks every 3 bytes into 4 characters from a 64-symbol alphabet (A–Z a–z 0–9 + /), padding the tail with =. It exists so binary data can ride through channels that only promise to preserve text — email bodies, JSON strings, data URIs — at the cost of 33% growth. It is an encoding, not encryption: anyone can reverse it.

Base64url & the JWT trap

URLs and filenames can't carry + or /, so base64url swaps them for - and _ and usually drops the = padding. JWTs are the famous case — paste a token segment into a strict decoder and it chokes. This page accepts both alphabets and missing padding, and shows the base64url form on its own row.

URL percent-encoding

Each byte outside the unreserved set (letters, digits, - _ . ~) becomes % plus two hex digits — one %XX per byte, so 💩 is four of them. Beware the space: RFC 3986 says %20, but HTML form encoding says + — this page treats + as a literal plus, so form-encoded data needs its pluses swapped first.

Hex

One byte, two hex digits — the most literal view of binary data short of bits. It's the lingua franca of digests, MAC addresses, and memory dumps. The decoder here is forgiving: spaces, 0x prefixes, colons and case are all tolerated; only an odd digit count or a non-hex character is an error.

Binary

The bits themselves, padded to eight per byte. Nobody transmits data this way — it's a teaching view, and the place to see why hex won: the same byte is 01001000 in binary, 48 in hex. The decoder accepts any spacing as long as the digit count is a multiple of eight.

SHA-256 & SHA-512

One-way fingerprints from the SHA-2 family, computed here with your browser's built-in WebCrypto. Fixed sizes — 64 and 128 hex digits — no matter the input, and changing a single bit scrambles the whole digest (edit one character above and watch). Unlike the Character Inspector's MD5 panel, these are still considered collision-resistant; unlike encryption, there is no way to "decode" a digest.

Encoding is not encryption

Base64, hex and friends require no key and hide nothing — they're transport formats. If a system "protects" data by Base64-ing it, the protection is a polite fiction. Encryption needs a secret; hashing destroys the data one-way; encoding just re-spells it.

When decoding fails

Each format fails its own way: Base64 with characters outside its alphabet or an impossible length (4n+1); hex with an odd number of digits; binary with a bit count that isn't a multiple of 8. And bytes that aren't valid UTF-8 still decode — the text row shows � replacement characters and says so, because the bytes themselves are still perfectly good bytes.

Why UTF-8 is the baseline

"Plain text" has to become bytes somehow, and this page — like essentially the whole modern web — uses UTF-8: ASCII stays one identical byte, everything else uses 2–4. The same text in UTF-16 would produce different bytes, different Base64, and different digests, which is why "hash of a string" is meaningless until someone names the encoding.
